# Phase 2: Electric Thinking -- Nudges, Email, Verification & Manager Dashboard

This is Phase 2 of 3. Phase 1 already built: authentication (Google/Microsoft/email), org setup, text-based assessment with Claude Opus scoring, results dashboard, admin panel, database with all tables, and the brand/design system. DO NOT rebuild or modify the auth system, database schema, or brand design. Build on top of what exists.

This phase adds: nudge generation, email delivery, skill verification, skill completion flow, manager dashboard, social proof, and re-engagement.

## 1. Electric Nudges -- Personalized Weekly Learning

Every week, each active user receives a personalized learning nudge via email (primary) and in-app (secondary).

**Nudge Generation:**
- Nudges are generated by calling the **Anthropic Claude API (Opus model)** using: the user's active skill (from user_skill_status where status = 'yellow'), their user context summary (from the assessments table context_summary field), and their AI platform (from users table)
- The generation is controlled by a "Nudge Voice Guide" -- the system prompt stored in the nudge_voice_guide table (already built in Phase 1 admin panel). This guide is passed as the system prompt to Claude when generating nudges.
- The Nudge Voice Guide contains:
  - Writing style rules (no em dashes, no "delve" or "landscape" or "testament", no Rule of Three, no sycophantic tone, no bold-header bullet lists, no synonym cycling)
  - Anti-AI-writing patterns
  - Tone direction and examples of approved nudge writing
  - Emotional framing guidance: connect to how skills FEEL, not just what they DO. "The moment you realize you'll never start from a blank page again" hits harder than "Learn to use AI for first drafts."
  - The Electric Thinking voice and philosophy
- Each nudge contains (structured as JSON in content_json):
  - **idea**: One core idea tied to the skill they're developing (ONE idea, not three)
  - **use_case**: One use case personalized to their actual role and work (using specific context from their assessment, not generic)
  - **action**: One specific action they can try in under 10 minutes -- NOT a copy-paste prompt, but a "Make These Words Your Own" instruction: describe what to do, what to say to their AI (in their own words), and what to look for in the response. The nudge gives the SHAPE of the action, not the exact words.
  - **reflection**: One reflection question to deepen the skill
  - **story**: A two-sentence story of someone in a similar role using this skill (concrete, specific)
  - **opener**: A pattern-breaking stat, observation, or question that creates a knowledge gap. Never "This week we're covering [skill name]."
- Nudges are generated in advance (scheduled job runs Sunday night for Monday delivery) and stored in the nudges table
- **Repetition prevention**: When generating a nudge, pass ALL previous nudges for that user+skill to the Claude prompt. Include a clear instruction: "Here are the previous nudges this user received for this skill. Generate something meaningfully different -- different opener, different use case, different action. Do not repeat angles, examples, or stories from previous nudges." Without this, Opus will pattern-match to similar content week after week and a user stuck on a skill for 6 weeks gets 6 versions of the same email.
- **Subject line generation**: Include subject line as part of the nudge generation prompt. The subject line should pull from the nudge's opener or reference the user's specific context. "Your weekly nudge" is death. "That thing you do every Monday morning? There's a faster way." is life. Store subject_line in the content_json alongside the other nudge components.
- Build a "Generate Nudge" button in the admin panel that lets admins manually trigger nudge generation for testing

**Delivery:**
- Email (primary channel): sent via **Resend API**. Set up Resend integration now.
- Build all email templates using **React Email** (react.email), which compiles React components into cross-client-compatible HTML. All emails use the Electric Thinking brand colors (from CSS custom properties), Tomorrow font for headings, Source Sans 3 for body.
- Nudge email includes: the full nudge content (opener, idea, use case, action, reflection, story) beautifully formatted, a "Mark This Skill Complete" button (links to the app), and an "Open in App" link. Think mini-magazine page, not corporate training email.
- In-app (secondary): nudges appear in the "My Learning" section of the dashboard, newest first. Users can read past nudges.
- **Single notification, not double**: Use Resend webhooks to track email opens. When the email is opened, mark email_opened = true in the nudges table. The in-app notification badge should check this field. If email was opened, don't show the in-app notification as "new." If they act in-app first, mark in_app_read = true and the email can reflect that via a "You've already seen this in the app" note. State syncs through the database.
- Delivery day is configurable per user (stored in users.nudge_day, default Monday).

**Re-engagement:**
- If a user hasn't opened a nudge (email_opened = false AND in_app_read = false) for their last 3 nudges: trigger a different email. Shorter, warmer, lower pressure. "No pressure. Here's one thing you could try in 2 minutes."
- After 6 weeks of inactivity (no nudge interactions): pause nudges automatically (set users.nudges_active = false) and notify the manager via email.

**Scheduled Jobs:**
- Weekly cron: Sunday night, generate nudges for all active users (nudges_active = true, has an active yellow skill). Call Claude Opus for each user. Store in nudges table. Monday morning (or user's configured day), send via Resend.
- Daily cron: check for re-engagement triggers (3 consecutive unopened nudges, 6 weeks inactive).
- Quarterly cron: send re-assessment reminder emails to users whose last assessment was 90+ days ago.
- All cron jobs should have error handling and logging. If a nudge fails to generate for one user, log the error and continue with the next user. Don't let one failure stop the batch.
- **Cron execution strategy for Replit**: Use `node-cron` (or equivalent scheduler library) inside a long-running background worker process. Replit Deployments support background workers alongside the web server. The worker checks schedules and runs jobs. Each job type stores a "last_successful_run" timestamp in system_config so: (a) the admin health dashboard can show when jobs last ran, (b) if the worker restarts, it can determine whether it missed a scheduled run and catch up, (c) jobs are idempotent (running the same job twice for the same period doesn't create duplicate nudges). Don't rely on external cron services.
- **API cost protection**: if a single nudge generation batch exceeds a configurable cost threshold (stored in system_config, default $50), pause the batch, send what's been generated so far, and email the system admin. Prevents runaway API costs from bugs or loops.
- **Time zone awareness**: Use the user's timezone field (from the users table, set during onboarding or in settings) to determine "Monday morning." Nudges should arrive between 8-9 AM in the USER's local time, not the server's. Group sends by timezone to batch efficiently. Default timezone: America/Los_Angeles.

## 2. Skill Verification & Completion

**Verification is always available, not gated behind nudges.** A user can click "Mark Skill Complete" on any Yellow skill from their dashboard at any time, whether or not they've received a nudge for that skill. Nudges are the teaching mechanism, but don't block verification behind them. This matters for speedrunners and for anyone who already knows the material. If someone clears Level 0 in one sitting because they already have those skills, the system should let them. The nudge system will automatically shift to their next active skill.

When a user clicks "Mark Skill Complete" (from nudge email button, in-app dashboard, or directly from a skill node):
- They're presented with 3 multiple-choice questions related to that skill
- Questions are generated by **Claude (Sonnet model)** based on the skill definition from the skills table, personalized to the user's context_summary from their assessment
- Questions reaffirm understanding, not trick the user. "Confirm you get this" not "gotcha."
- Store questions and answers in the verification_attempts table

**Passing (2/3 or 3/3 correct):**
- Skill status changes from 'yellow' to 'green' in user_skill_status
- Set completed_at timestamp
- Celebration animation on screen (confetti, glow, satisfying visual moment)
- Small pride message: "Nice. Context Setting: locked in."
- The next skill in sequence (by sort_order within the level) becomes 'yellow' (active)
- Future nudges shift to the new active skill
- Create a badge record (badge_type = 'skill_complete')
- Create an activity_feed record (event_type = 'skill_complete')
- Send a skill completion email via Resend: celebration, preview of what's next

**Failing (0/3 or 1/3):**
- Skill stays 'yellow'
- Encouraging message: "Not quite. Revisit this week's nudge and try again when you're ready."
- No penalty, no cooldown -- they can retry immediately

**Level Up (all 5 skills in a level are 'green'):**
- BIG celebration moment. Special animation, visual fanfare, confetti explosion.
- Identity upgrade message: "You're now a Thought Partner."
- The next level's skills become visible/unlocked on the dashboard
- Send a level-up email via Resend with a shareable badge image (generate a simple branded card: "I'm a Level 2 AI Thought Partner, certified by Electric Thinking" with ET brand colors and wordmark). The badge should be an image users can download or share on LinkedIn/Slack.
- Create badge record (badge_type = 'level_up')
- Create activity_feed record (event_type = 'level_up')
- Level badge added to user's profile/dashboard

**When ALL levels are complete:** Ultimate celebration. Unique, special moment. Worth screenshotting and sharing.

**Streak tracking:** Track consecutive weeks where a user interacts with a nudge or completes a skill. Store in user_skill_status.streak_count. Award streak badges at 3-week, 5-week, 10-week milestones.

**Speedrun detection (informational, not blocking):**
- Never prevent a user from completing skills as fast as they want. If someone genuinely knows the material, let them prove it and move on.
- BUT: if a user completes 5+ skills in a single day, or jumps 2+ levels in a single week, send the manager an informational notification: "[Name] just completed all Level 1 skills in one sitting. They might already know this material, or it might be worth a quick check-in."
- Log these events in the activity_feed so the admin analytics can track how many people speedrun vs. pace normally.
- The assessment might score someone as Red on skills they actually have. Blocking them from proving they know it would be punishing them for a scoring error.

## 3. Manager Dashboard

Add a dedicated dashboard view for users with manager or org_admin role:

- **Team Overview:** Visual distribution showing how many team members are at each level (0-4). Bar chart or similar. Clear, visual, at-a-glance.
- **Individual View:** Click into any team member to see their full skill profile (Red/Yellow/Green for every skill) and their progression timeline (when they completed each skill, when they leveled up).
- **Progression Tracking:** Timeline showing when each person leveled up or completed skills. Track momentum over time.
- **Team Gaps:** Highlight the most common Red skills across the team -- helps managers know where live training should focus.
- **Nudge Control:** Toggle to pause/resume nudges for any individual team member. Simple on/off switch that sets users.nudges_active.
- **Social Proof Moments:** Activity feed showing when team members level up or complete skills (from activity_feed table).
- **Export:** Export team progress as PDF report or CSV data.
- Managers do NOT approve individual nudges. The system is autonomous.
- **Manager visibility boundary:** Managers see skill scores, levels, and progression timelines. They do NOT see assessment transcripts, user context summaries, or nudge content.

**Org-Level Settings (configurable by org admin):**
- Default nudge day for the org (users can still override individually)
- Social proof anonymization: show teammate names in activity feed, or anonymize ("A teammate just hit Level 2")
- Custom welcome message: text that appears on the team's landing page and invite emails ("Welcome to [Company]'s AI fluency program. Here's what to expect...")
- Assessment framing: "optional" (take it when you're ready) vs. "expected" (your manager has enrolled you). Changes the tone of invite emails and dashboard prompts.

**Manager Onboarding:** When someone is first assigned the manager role, trigger a 3-email welcome sequence via Resend:
1. How the dashboard works
2. How to talk to your team about the assessment without it feeling like surveillance

## 5. Email System

All emails sent via **Resend API** using **React Email** templates. Every email uses the Electric Thinking brand (warm cream backgrounds, magenta CTAs, Tomorrow headings, Source Sans 3 body, the ELECTRIC THINKING wordmark).

**Email Sender Configuration:**
- FROM address: "Electric Thinking <hello@electricthinking.ai>" -- friendly, recognizable, not "noreply"
- Reply-to: "support@electricthinking.ai" -- if someone replies to a nudge, it should go somewhere real (configure this in Resend or forward to Devin's inbox)
- Store FROM and reply-to in system_config so they're changeable without code
- Resend requires DNS verification of the sending domain (electricthinking.ai). Add SPF, DKIM, and DMARC records to Cloudflare. This is critical for deliverability -- without it, nudge emails land in spam and the product silently breaks.

**Email Bounce Handling:**
- Integrate Resend's bounce and complaint webhooks. When a hard bounce occurs (invalid address, domain doesn't exist), automatically set a flag on the user record (email_valid = false) and stop all future email sends to that address. Don't silently keep sending to dead addresses, it destroys your sender reputation and eventually ALL emails start landing in spam for everyone.
- Soft bounces (full mailbox, temporary failure): retry up to 3 times over 48 hours. If still failing, pause email for that user and flag for admin review.
- Spam complaints (user marks email as spam in their email client): immediately set nudges_active = false and stop all email. Log the complaint. This is a CAN-SPAM requirement.
- Show bounce/complaint counts in the admin system health view so the admin can monitor deliverability.

Build these email templates:
- **Invite email** -- when a manager invites someone. Warm, explains what they're joining.
- **Welcome email** -- after assessment completion. Summarizes their level, skill priorities, what to expect. Identity framing: "You're a [Level Name]."
- **Weekly nudge email** -- the core product touchpoint. Beautifully formatted nudge content.
- **Skill complete email** -- celebration. Preview of what's next.
- **Level up email** -- bigger celebration with shareable badge image.
- **Re-engagement email** -- different tone for users quiet 3+ weeks. Shorter, warmer, lower pressure.
- **Re-assessment reminder** -- at 90 days, suggesting they re-take.
- **Manager onboarding sequence** -- 3 emails over the first week.
- **Abandoned assessment email** -- sent 24 hours after someone starts but doesn't finish the assessment. "You started your AI fluency assessment but didn't finish. It only takes a few more minutes. Pick up where you left off." Direct link to resume.
- All emails include an unsubscribe link.

**Email Notification Preferences (user-facing):**
- Build a notification preferences page (linked from user settings and from the unsubscribe link in every email). Users control which email types they receive with separate toggles:
  - "Weekly learning nudges" (on/off)
  - "Progress celebrations" (skill complete, level up emails) (on/off)
  - "Assessment reminders" (re-assessment, abandoned assessment) (on/off)
  - "Turn off all emails" -- one-click global unsubscribe for CAN-SPAM compliance. This sets ALL toggles to off. Show a confirmation: "You won't receive any emails from Electric Thinking. You can still use the app and re-enable emails anytime."
- The unsubscribe link in every email should go to this preferences page (pre-authenticated via a secure token in the URL so the user doesn't have to log in). One-click unsubscribe header (List-Unsubscribe) should also be set in email headers for Gmail/Outlook compliance.
- When a user has all emails turned off, their dashboard should show a subtle banner: "You've turned off email notifications. You can re-enable them in settings."

Update the admin panel Email Templates section (placeholder from Phase 1) to let admins preview and edit these templates.
