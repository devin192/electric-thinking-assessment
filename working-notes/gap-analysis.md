## 1. Authentication (Google OAuth, Microsoft OAuth)

**Status: Not Implemented**

The spec requires three auth methods: Google OAuth, Microsoft OAuth (Azure AD), and email/password. The codebase only has email/password auth via bcryptjs + express-session.

**Evidence:**
- `/Users/devin/electric-thinking-assessment/server/auth.ts`: Only configures session-based auth with bcryptjs. No OAuth providers.
- `/Users/devin/electric-thinking-assessment/client/src/pages/auth.tsx`: Login and Register forms only show email/password fields. No "Sign in with Google" or "Sign in with Microsoft" buttons.
- `package.json` includes `passport` as a dependency but it is not imported or used anywhere in the server code.

**Impact:** Enterprise clients (Disney, Intuit, Visa) will expect SSO. This is a major gap for enterprise readiness. Without OAuth, every user needs to create a new password, which creates friction and doesn't meet typical enterprise IT requirements.

## 3. Assessment Scoring (AI Model)

**Status: Implemented But Misses Intent**

The spec and Final Replit Push both reference using Claude Opus for scoring ("Generated by Claude Opus during the scoring step"). The codebase uses claude-sonnet-4-20250514 for all AI calls.

**Evidence:**
- `/Users/devin/electric-thinking-assessment/server/assessment-ai.ts` line 6: `const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });`
- Same file, `scoreAssessment()` function uses `model: "claude-sonnet-4-20250514"` (line visible in scoring call).
- `getAssessmentResponse()` also uses claude-sonnet-4-20250514.
- `/Users/devin/electric-thinking-assessment/server/nudge-ai.ts` also uses claude-sonnet-4-20250514.

**Impact:** Sonnet is faster and cheaper but the spec explicitly calls for Opus-level quality on scoring. Scoring is the highest-stakes AI call (determines someone's level placement, generates their bright spots, future self, signature skill). The quality difference between Sonnet and Opus matters most here. Chat messages during the assessment are fine on Sonnet.

**Recommendation:** Switch `scoreAssessment()` to use Claude Opus. Keep `getAssessmentResponse()` and nudge generation on Sonnet.

## 6. RPG Skill Map (Council Top 5 Move #2)

**Status: Implemented But Misses Intent**

The Final Replit Push says: "I want this to be genuinely impressive. Push the visual design hard. Think Duolingo's skill tree meets a fantasy RPG map. If the first version looks like a chart with better colors, that's not enough. It should make someone say 'whoa' when they see it."

**Evidence:**
- `/Users/devin/electric-thinking-assessment/client/src/components/rpg-map.tsx` (428 lines): Implements an SVG-based map with nodes, paths, fog of war, level regions, popovers, and Framer Motion animations.
- Three node states: green (checkmark + glow), yellow (pulsing), red (dimmed).
- Fog of war gradient on levels beyond current+1.
- Bezier curve paths connecting nodes.
- Level regions with colored backgrounds and labels.
- Click interaction with popover showing skill details.

**What's Missing vs. the spec:**
1. **"Nodes are destinations, not dots"**: The spec says each node should be "a small illustrated icon or badge, not a circle." The current implementation uses circles (filled/outlined) with checkmarks for green nodes. They look like chart nodes, not destinations.
2. **"Regions, not rows"**: The spec describes distinct visual regions ("starting village, open road, workshop/forge, command center"). The current implementation has labeled rectangles with light background fills. They don't evoke the sense of place the spec describes.
3. **"You are here" indicator**: The spec calls for "a glowing marker or character icon on your current active skill that pulses." The current yellow nodes pulse, but there's no distinct "you are here" marker differentiating the ACTIVE skill from other yellow skills.
4. **Level-up animation**: The spec says "fog dramatically lifts on the next region with an animation." The fog of war is static (CSS gradient). No dynamic fog-lifting animation exists.
5. **Exploration of fogged nodes**: The spec says clicking fogged nodes should show "a mysterious silhouette with 'Keep going to discover this skill.'" The current implementation does not allow clicking on fogged nodes.
6. **Mobile experience**: The spec notes "responsive (works on mobile too, perhaps as a vertical scroll on small screens)." The current SVG has a fixed 600px viewBox. On mobile, it shrinks to fit but doesn't reflow to a vertical layout.

**Impact:** The map is functional and has good bones (paths, fog, color coding, animations). But it reads as "a chart with better colors" rather than the "whoa" moment the spec demands. The council review reinforces this: "Make the skill map literally a game map, not a chart. Terrain, landmarks, a character standing on a path."

## 14. Auto-Save Assessment Progress

**Status: Not Implemented**

The spec says: "Auto-save assessment progress so if connection drops mid-conversation, the user can resume where they left off."

**Evidence:**
- `/Users/devin/electric-thinking-assessment/server/routes.ts`: POST /api/assessment/:id/message saves each message to the assessment transcript.
- However, there is no resume mechanism. If the user navigates away mid-assessment, GET /api/assessment/active returns the in-progress assessment, but the client-side state (chat history, voice connection) is lost.
- The assessment page does not load previous messages from the transcript on mount. It starts fresh.
- The 24-hour abandoned assessment check in cron.ts sends a reminder email, but that email links back to start a new assessment, not resume the existing one.

**Impact:** A connection drop during a 10-15 minute voice assessment means starting over. This is frustrating for the user and wasteful of their time. The transcript IS being saved server-side, so resume is technically possible with client-side work.
