# Mobile Voice Audit Report

Generated by Medic agent. Systematic audit of the voice assessment flow on mobile devices, covering connection, audio playback, mic capture, encoding, UI responsiveness, and touch targets.

---

## Issues Found and Fixed

### 1. Stale Mute Closure (Audio Capture Bug)

**CATEGORY:** Mic Capture
**SYMPTOM:** Toggling mute during a voice assessment had no effect. The user could tap the mute button but audio continued being sent to ElevenLabs.
**ROOT CAUSE:** The `onaudioprocess` callback inside `connectVoice` captured `isMuted` from the `useCallback` closure. Because `isMuted` was a React state variable and `useCallback` listed it as a dependency, the entire `connectVoice` function would re-create when mute toggled, but the ScriptProcessor callback was already bound to the old WebSocket's `onopen` handler. The callback's reference to `isMuted` was frozen at the value it had when the WebSocket connection was established.
**FIX:** Added `isMutedRef` (a React ref) that stays in sync with the `isMuted` state via a `useEffect`. The `onaudioprocess` callback reads from the ref instead of the closure. Also removed `isMuted` from the `useCallback` dependency array to prevent unnecessary reconnection cycles.
**FILE:** `client/src/pages/assessment.tsx`
**REGRESSION RISK:** None. Desktop behavior is identical.

### 2. No WebSocket Reconnection (Cellular Network Resilience)

**CATEGORY:** Connection
**SYMPTOM:** On cellular networks, the WebSocket connection could drop mid-conversation with no recovery. The user would see a blank screen or frozen state.
**ROOT CAUSE:** The `ws.onclose` handler saved the transcript and stopped, with no reconnection logic. Cellular networks are aggressive about dropping idle or long-lived WebSocket connections, especially during network handoffs (WiFi to cellular, cell tower handoff).
**FIX:** Added automatic reconnection with exponential backoff (1s, 2s, 4s) for up to 3 attempts. Only abnormal closures trigger reconnection (codes other than 1000 and 1001). Intentional disconnects (user tapping "End Conversation") set the retry counter to max to prevent reconnection. The `reconnectAttemptsRef` counter resets on successful connection.
**FILE:** `client/src/pages/assessment.tsx`
**REGRESSION RISK:** None. Normal close events (code 1000/1001) are unaffected.

### 3. Mic Stream Not Cleaned Up on Normal Close

**CATEGORY:** Mic Capture
**SYMPTOM:** After a voice conversation ended normally (WebSocket closed with code 1000), the microphone remained active. On iOS, this shows a persistent red recording indicator in the status bar and drains battery.
**ROOT CAUSE:** The `ws.onclose` handler called `saveTranscript()` but did not stop the MediaStream tracks. The mic cleanup only happened in `disconnectVoice()` (user-initiated) and `ws.onerror` (error path), not on normal WebSocket closure.
**FIX:** Added `mediaStreamRef.current.getTracks().forEach(t => t.stop())` to the `ws.onclose` handler after the reconnection check. Only cleans up if no reconnection will be attempted.
**FILE:** `client/src/pages/assessment.tsx`
**REGRESSION RISK:** None.

### 4. AudioContext Suspended After Background/Interruption

**CATEGORY:** Audio Playback
**SYMPTOM:** On iOS, if a phone call comes in, the user opens Control Center, or the app is briefly backgrounded during a voice assessment, audio playback silently stops. The AI continues responding but the user hears nothing.
**ROOT CAUSE:** iOS suspends the AudioContext when the app loses audio focus. The `playAudioChunk` function did not check or recover the AudioContext state before attempting playback.
**FIX:** Added a `ctx.state === "suspended"` check at the top of `playAudioChunk`. If suspended, `ctx.resume()` is called before decoding and scheduling audio. Also added the same check in `connectVoice` before establishing the WebSocket, since the AudioContext may have been suspended during the token fetch.
**FILE:** `client/src/pages/assessment.tsx`
**REGRESSION RISK:** None. On desktop, `ctx.state` is always "running".

### 5. getUserMedia Without Audio Processing Constraints

**CATEGORY:** Mic Capture
**SYMPTOM:** On mobile devices with built-in speakers close to the microphone (especially phones without earbuds), the captured audio could include echo from the AI's speech, creating feedback.
**ROOT CAUSE:** `getUserMedia` was called with `{ audio: true }`, which uses browser defaults. Many mobile browsers default to echo cancellation off or minimal.
**FIX:** Changed constraints to `{ audio: { echoCancellation: true, noiseSuppression: true, autoGainControl: true } }`. These are hints, not guarantees, but most mobile browsers honor them.
**FILE:** `client/src/pages/assessment.tsx`
**REGRESSION RISK:** None. Desktop browsers also support these constraints.

### 6. Shared AudioContext/MediaStream Not Validated

**CATEGORY:** Audio Playback / Mic Capture
**SYMPTOM:** If the user sat on the warmup page for a long time (e.g., reading the instructions carefully, incoming phone call), the pre-initialized AudioContext could be closed or the MediaStream tracks ended by the time they started the assessment. The assessment would fail silently or produce no audio.
**ROOT CAUSE:** `getSharedAudioContext()` and `getSharedMediaStream()` returned the cached objects without checking their state. A closed AudioContext or ended MediaStream is unusable.
**FIX:** Added validation in both getter functions. `getSharedAudioContext()` now returns `null` if the context state is "closed". `getSharedMediaStream()` returns `null` if all audio tracks have ended. The assessment page's `connectVoice` already has a fallback path for when these return `null` (creates fresh instances).
**FILE:** `client/src/lib/audio-context.ts`
**REGRESSION RISK:** None. The fallback path already existed.

### 7. Touch Targets Below 44px Minimum

**CATEGORY:** Touch Targets
**SYMPTOM:** Small buttons and text links on the assessment page were difficult to tap on mobile, especially the "Having trouble with live voice?" text link and the send button.
**ROOT CAUSE:** Several interactive elements relied on their text content for sizing, which fell below the 44x44px Apple HIG minimum. The "Having trouble?" link was approximately 14px tall. The send icon button was the shadcn default size.
**FIX:** Added `min-h-[44px] min-w-[44px]` to all buttons across all three assessment modes (full-duplex, voice-to-text, text-only). Text links got `min-h-[44px] px-4 inline-flex items-center` to create a large enough tap target without changing visual appearance. Increased gap between mute and end-call buttons from `gap-3` (12px) to `gap-4` (16px) for the 8px+ spacing requirement.
**FILE:** `client/src/pages/assessment.tsx`
**REGRESSION RISK:** None. The buttons are slightly larger on desktop, which is fine.

### 8. Dialog Not Scrollable on Small Screens

**CATEGORY:** UI Responsiveness
**SYMPTOM:** The "Switch to text-only?" confirmation dialog could extend below the viewport on short phones (e.g., iPhone SE), making the buttons unreachable.
**ROOT CAUSE:** The DialogContent had no max-height or overflow handling.
**FIX:** Added `max-h-[90vh] overflow-y-auto` to the DialogContent.
**FILE:** `client/src/pages/assessment.tsx`
**REGRESSION RISK:** None.

### 9. Missing touch-action: manipulation

**CATEGORY:** Touch Targets
**SYMPTOM:** 300ms tap delay on some older mobile browsers. Double-tap-to-zoom could accidentally trigger on buttons.
**ROOT CAUSE:** While modern browsers eliminate the 300ms delay when a proper viewport meta tag is set (which this app has), `touch-action: manipulation` is the CSS-level guarantee that double-tap-to-zoom is disabled on interactive elements.
**FIX:** Added a global CSS rule applying `touch-action: manipulation` to all `button`, `a`, `input`, `textarea`, `select`, and `[role="button"]` elements.
**FILE:** `client/src/index.css`
**REGRESSION RISK:** None. This is standard mobile web practice.

### 10. iOS Input Zoom Prevention

**CATEGORY:** UI Responsiveness
**SYMPTOM:** When tapping into the text input field on iOS, Safari auto-zooms the page if the input font size is below 16px. The page then stays zoomed in, requiring manual pinch-to-zoom-out.
**ROOT CAUSE:** iOS Safari zooms to make text readable if the font-size is below 16px. The base font was 15px.
**FIX:** Added a `@media (max-width: 768px)` rule setting `font-size: 16px` on body, input, textarea, and select elements.
**FILE:** `client/src/index.css`
**REGRESSION RISK:** None. Only affects screens 768px and below.

## Items Already Correct (No Fix Needed)

### Viewport Meta Tag
`<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1">` is already correct. The `maximum-scale=1` prevents pinch-to-zoom, which is appropriate for a PWA-style assessment experience.

### AudioContext Created Inside User Gesture
The warmup page creates the AudioContext synchronously inside the button click handler before any async operations. This satisfies iOS's autoplay policy. The assessment page's fallback path (no shared context) also creates the AudioContext synchronously in `connectVoice`, which is triggered from a `useEffect` following user navigation, which iOS also allows.

### Audio Playback Scheduling
The `playAudioChunk` function uses `currentTime`-based scheduling (`Math.max(now, nextPlayTimeRef.current)`) to prevent overlapping chunks. This is the correct approach.

### PCM16 Encoding
The downsampling and encoding logic is correct: `Math.floor(i * ratio)` for sample selection, `Math.max(-32768, Math.min(32767, sample * 32768))` for clamping, and the result is sent as base64-encoded little-endian Int16Array. ElevenLabs expects this format.

### ElevenLabs Ping/Pong
The WebSocket message handler responds to `ping` events with `pong` responses, which serves as a keep-alive mechanism. This is the ElevenLabs protocol's built-in heartbeat.

### Horizontal Overflow
No horizontal scroll issues found. The layout uses `flex`, `max-w-2xl`, and percentage-based widths that fit within the viewport.
