Generated by Wordsmith agent. Documents all changes to the system prompts in assessment-ai.ts and nudge-ai.ts.

## assessment-ai.ts: Scoring Prompt

### Change 1: Explicit Red/Yellow/Green Scoring Criteria

**Before:** Scoring criteria were vague: `"green" (demonstrated), "yellow" (emerging/partial), "red" (not demonstrated)`.

**After:** Each status now has a precise, behavioral definition:
- **Green**: User described it as a regular, habitual part of their workflow AND gave at least one specific example.
- **Yellow**: User mentioned trying it, described inconsistent use, or showed awareness without clear evidence of regular practice.
- **Red**: User never mentioned it, explicitly said they don't do it, or no evidence in the transcript. When in doubt, choose red.

**Why:** The prompt-architect skill identifies "generous scoring" as a common failure mode. Vague criteria let the model give yellow for anything vaguely mentioned. Explicit criteria force the model to look for evidence of actual habitual use before going green.

### Change 2: Richer contextSummary Instructions

**Before:** `"Generate a context summary of the user: their role, work description, AI platform, specific examples mentioned, communication style, what excites/frustrates them"`

**After:** Expanded instructions that emphasize capturing specific details from the conversation, with an explicit note that generic summaries are a failure. The contextSummary powers all downstream personalization (challenges, verification questions), so thin context produces generic content.

**Why:** The prompt-architect skill notes the scoring-to-challenge handoff: "If context_summary is thin or generic, challenges will be generic too. A bad scoring prompt creates downstream quality problems."

### Change 3: Writing Style Constraints for Generated Text

**Before:** No constraints on the writing style of brightSpotsText, futureSelfText, or signatureSkillRationale.

**After:** Added explicit WRITING STYLE section banning em dashes, all CLAUDE.md banned words, sycophantic openers, and requiring specific references to things the user actually said. Emphasizes progress framing (what they've accomplished, not what's left).

**Why:** The scoring prompt generates text the user reads directly (bright spots, future self, signature skill rationale). Without style constraints, the model defaults to its natural patterns, which include em dashes, "delve," and congratulatory openers. These clash with the product voice.

### Change 4: Em Dash Removal from Examples

**Before:** Three example strings in the prompt contained em dashes.

**After:** All em dashes replaced with periods.

**Why:** The model imitates examples. If examples contain em dashes, generated text will too. This is the CLAUDE.md #1 hard ban.

### Change 5: Per-Skill Explanation Requirement

**Before:** `"write a one-sentence explanation of WHY it got that score"`

**After:** `"write a one-sentence explanation of WHY it got that score, referencing specific evidence from the transcript"`

**Why:** Forces the model to cite evidence rather than generate vague explanations like "The user showed some awareness of this skill."

## nudge-ai.ts: Challenge Generation Prompt

### Change 1: Personalization Enforcement

**Before:** `"ONE use case personalized to their actual role and work context"`

**After:** Added explicit instruction: "Reference something specific from their assessment context (their job, their tools, their examples). If the use case could apply to anyone, rewrite it."

**Why:** Generic use cases are the #1 quality failure in challenges. The user context is available but the model can ignore it if not pushed.

### Change 2: Story Specificity

**Before:** `"A two-sentence story of someone in a similar role using this skill (concrete, specific)"`

**After:** Added: `"Be concrete: include a specific role, a specific situation, and a specific outcome. 'A content lead at a streaming service' not 'A marketing manager.'"`

**Why:** The prompt-architect skill calls out "Stories that feel made up or too perfect" as a watch-for. Forcing specific details makes stories more believable.

### Change 3: Anti-Sycophantic Opener

**Before:** Only instruction was about pattern-breaking openers.

**After:** Added explicit: `"Never open with a compliment like 'Great job' or 'You're doing amazing.'"`

**Why:** The model's default opener behavior leans congratulatory. The CLAUDE.md writing rules ban sycophantic openers.

### Change 4: Action Step Constraint

**Before:** `"NOT a copy-paste prompt"`

**After:** Added: `"If it requires setup or multiple tools, it's too big."`

**Why:** The prompt-architect skill notes "Set up a custom GPT is not 10 minutes for a beginner" as a common 10-minute constraint violation.

### Change 5: Writing Rules Block

**Before:** Writing constraints only existed in the system prompt fallback (when no voice guide is stored in DB).

**After:** Added a full WRITING RULES block directly in the user prompt, covering em dashes, banned words, "It's not just X it's Y" constructions, Rule of Three, and sentence length variation.

**Why:** The voice guide stored in the DB may not contain these constraints. Adding them to the user prompt ensures they always apply regardless of what the voice guide says.

### Change 6: Subject Line Update

**Before:** `"NOT 'Your weekly nudge.'"`

**After:** `"NOT 'Your weekly challenge.'"`

**Why:** Naming consistency. Even though the subject_line field name stays the same, the example text should match the user-facing terminology.

## nudge-ai.ts: Verification Question Prompt

### Change 1: Tone Reframe

**Before:** `"Generate 3 multiple-choice verification questions"`

**After:** `"Generate 3 multiple-choice skill check questions for this AI fluency skill. These confirm understanding, they don't trick people. The tone is encouraging, like a coach checking in, not a professor giving a test."`

**Why:** The naming rules say "verification" becomes "skill check" in user-facing contexts. The prompt sets the tone for the generated content.

### Change 2: Question Framing

**Before:** No guidance on question framing.

**After:** Added: `"Use 'Which of these would work best for...' framing, not 'Which of the following is correct?'"`

**Why:** The prompt-architect skill specifies this exact pattern. It makes questions feel less like a test and more like practical scenario evaluation.

### Change 3: Explanation Tone

**Before:** `"Brief explanation of why this is correct"`

**After:** `"Brief, encouraging explanation"` plus guidance: `"The explanation should be encouraging and teach something. 'This works because...' not 'The correct answer is A because...'"`

**Why:** After a user passes, the explanation reinforces learning. It should feel like a teaching moment, not a grading rubric.

### Change 4: Anti-Pattern Rules

**Before:** No explicit rules against common bad patterns.

**After:** Added rules against "All of the above" / "None of the above" options, and requirement that wrong answers be "plausible but clearly inferior if you understand the skill."

**Why:** The prompt-architect skill lists these as explicit watch-fors. They reduce question quality and frustrate users.

### Change 5: Practical Testing Focus

**Before:** `"The questions should REAFFIRM understanding"`

**After:** `"Each question tests whether they understand the skill in practice, not whether they can define it."`

**Why:** Definition-based questions (e.g., "What is Context Setting?") test memorization. Practice-based questions ("Which approach would work best when...") test understanding.
